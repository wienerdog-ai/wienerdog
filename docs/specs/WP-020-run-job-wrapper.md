# WP-020: Implement the run-job wrapper (clean env, TCC-guard, watchdog, logs, fail-loud, catch-up)

## Context (read this, nothing else)

When the OS scheduler fires (a launchd plist on macOS, a systemd user timer on Linux —
both generated by WP-013), it launches exactly one command: `wienerdog run-job <name>`.
**This work package builds `run-job`** — the short-lived wrapper that turns a raw
OS-scheduled launch into a safe job execution: it builds a clean environment, refuses
to touch macOS TCC-protected paths, runs the job's "brain" under a hard watchdog,
tees output to rotated logs, on failure alerts the user loudly, and records a
`last_success` watermark. It also implements `run-job --catch-up`, which runs jobs
that were missed while the machine was powered off.

Everything here is codified from the owner's claude-os production system, where these
exact failures were learned the hard way:

- **launchd/systemd children inherit almost nothing.** A scheduled process starts with
  a nearly empty environment — no useful `PATH`, sometimes no `HOME`. If `run-job`
  spawns `claude`/`node` assuming an interactive shell's env, it fails silently. So
  `run-job` **builds a clean env explicitly** (L5/L6 lessons): an absolute `PATH` that
  includes the node and `claude`/`codex` binary locations, and `HOME`.
- **macOS TCC-protected paths hang unattended jobs forever.** A launchd job that reads
  a file under `~/Desktop`, `~/Documents`, `~/Downloads`, or iCloud
  (`~/Library/Mobile Documents`) blocks on an invisible permission prompt no one can
  answer — the infamous 4-hour hang. So `run-job` runs a **TCC-guard** first and
  **refuses** (fail-loud) rather than hang.
- **launchd `StartCalendarInterval` runs missed jobs on wake but NOT after a
  power-off.** So a laptop shut at 03:30 never dreams. The fix is a **catch-up** check
  (WP-013 registers a login + hourly catch-up plist on macOS that calls
  `run-job --catch-up`) which compares each job's `last_success` to its schedule and
  runs anything overdue. On Linux, systemd `Persistent=true` catches up natively, so
  catch-up is a no-op there.

Product invariants that govern every line here:

- **Nothing outlives the job (ADR-0004).** `run-job` spawns the brain as a detached
  child (its own process group), waits under a watchdog, and on timeout **kills the
  whole process tree** and exits. No daemon, no polling loop, no listener, no
  telemetry. The watchdog must guarantee the child and the timer are gone before
  `run-job` returns, on both the success and the timeout path.
- **Fail-loud, never silent (Threat model T6).** On failure or timeout, `run-job`
  alerts the user — an email via `wienerdog gws _alert` if Google is configured (a
  built-in self-send to the user's own address; WP-018), else a red banner line
  prepended to the injected session digest (`state/digest.md`) so the next session
  surfaces it. It never swallows a failure.
- **The dream job has no send access (ADR-0007).** `run-job`'s fail-loud alert is the
  one built-in self-send; it never grants or widens any send capability, and a skill
  job that sends does so only under an existing grant (enforced by `gws`, not here).

### Exact contracts

#### `src/scheduler/tccguard.js`

```js
/** macOS TCC-protected home-relative prefixes that hang unattended jobs.
 *  Exported so tests and doctor can reference the same list. */
const TCC_PREFIXES = ['Desktop', 'Documents', 'Downloads',
  'Library/Mobile Documents'];   // last = iCloud Drive root

/** Is `p` inside a TCC-protected location (macOS only)?
 *  @param {string} p           an absolute path (realpath-resolved by caller if a symlink)
 *  @param {string} home        the user's home dir
 *  @param {NodeJS.Platform} [platform=process.platform]
 *  @returns {{protected:boolean, prefix:string|null}}
 *  Non-darwin platforms are never protected → {protected:false, prefix:null}.
 *  On darwin: protected iff `p` equals or is under home/<one of TCC_PREFIXES>
 *  (compare on path segments, not string prefix — 'Documents' must not match
 *  'DocumentsArchive'). */
function checkPath(p, home, platform = process.platform)

/** Guard a set of paths a job will touch. Returns the first offender or ok.
 *  @param {string[]} paths @param {string} home @param {NodeJS.Platform} [platform]
 *  @returns {{ok:boolean, offending:string|null, prefix:string|null}} */
function guard(paths, home, platform = process.platform)

module.exports = { TCC_PREFIXES, checkPath, guard };
```

`run-job` calls `guard([vaultDir, cwd], home)` before spawning the brain. On darwin, if
the vault or the job's working dir is under a protected prefix, the job is refused (a
fail-loud, not a hang). The default `~/wienerdog` vault and `~/.wienerdog` core are both
under `$HOME` root (not protected) — safe. An adopted vault a user placed under
`~/Documents` trips the guard, which is the correct behavior (ARCHITECTURE: adopted
vaults in TCC paths get a doctor warning; this is the runtime backstop).

#### `src/cli/run-job.js`

```js
/** wienerdog run-job <name>        run one job now
 *  wienerdog run-job --catch-up    run every job overdue vs its schedule
 *  Exit 0 = success or "nothing overdue". Exit 1 = the job failed / timed out /
 *           was refused by the TCC-guard (WienerdogError, after fail-loud).
 *  @param {string[]} argv @returns {Promise<void>} */
async function run(argv)
```

**`run-job <name>` pipeline (in order):**

1. `paths = getPaths()`. `job = require('../scheduler/jobs').findJob(paths, name)`;
   null → `WienerdogError('unknown job: <name>')`.
2. Resolve `vaultDir` from `config.yaml` (`vault:` line; reuse the existing minimal
   reader — `require('../core/dream/config').readDreamConfig(paths.config).vault`, which
   already throws if no vault). Set the job's working dir `cwd = vaultDir`.
3. **TCC-guard**: `const g = tccguard.guard([vaultDir, cwd], paths.home)`. If
   `!g.ok` → **fail-loud** (below) with reason
   `"refused: <g.offending> is under a macOS protected folder (<g.prefix>) — move the
   vault to ~/wienerdog"` and then `WienerdogError` (exit 1). Do NOT spawn.
4. **Clean env** — build explicitly (do not inherit `process.env` wholesale):
   ```js
   const env = {
     HOME: paths.home,
     PATH: [ path.dirname(process.execPath),      // node
             '/opt/homebrew/bin', '/usr/local/bin', // common claude/codex install dirs
             '/usr/bin', '/bin', '/usr/sbin', '/sbin' ].join(':'),
     WIENERDOG_JOB: name,                          // WP-018's send resolves routine from this
   };
   // Pass through only the few Wienerdog/harness overrides if already set:
   for (const k of ['WIENERDOG_HOME','WIENERDOG_VAULT','CLAUDE_CONFIG_DIR',
                    'CODEX_HOME','ANTHROPIC_API_KEY']) {
     if (process.env[k]) env[k] = process.env[k];
   }
   ```
   `WIENERDOG_JOB=<name>` matters: a `skill:` job that calls `gws gmail send` without an
   explicit `--routine` resolves its routine from `process.env.WIENERDOG_JOB`, so grant
   matching works headlessly (WP-018 contract).
5. **Choose the command from `job.run`:**
   - `builtin:dream` → `argv = [wienerdogBin(), 'dream', '--yes']`, run with `nodePath()`.
   - `builtin:<other>` → `WienerdogError('unknown builtin job: <other>')` (dream is the
     only built-in in v1).
   - `skill:<skill-name>` → run the harness headlessly on that skill: `cmd = 'claude'`,
     with args `['-p', '/<skill-name>']` (the slash-prefixed skill invocation, e.g.
     `/wienerdog-daily-digest`). Claude is the v1 default headless brain for routines;
     Codex-exec routines are a later WP — note this in "Decisions made".
6. **Logs**: `logDir = <paths.logs>/<name>` (mkdir -p). Open a per-run log file
   `<logDir>/<runStamp>.log` (`runStamp` = the resolved date-time, filesystem-safe).
   Tee the child's stdout+stderr to it (pipe the child streams to the write stream; also
   let them inherit is not enough for tee — pipe + write). After the run, **rotate**:
   keep only the newest 14 `*.log` files in `logDir`, delete the rest.
7. **Watchdog** (reuse `dream.js`'s pattern): spawn the child **detached** (own process
   group) with the clean `env` and `cwd`. Race its exit against
   `setTimeout(job.timeoutMinutes*60_000)`. On timeout: if alive,
   `process.kill(-child.pid, 'SIGKILL')` (whole tree), then treat as failure
   (`"job <name> timed out after N min"`). On non-zero exit: failure
   (`"job <name> exited N"`). Always clear the timer; never leave a child or timer
   behind.
8. **On success** (`code === 0`): `writeScheduleState(paths, name, {last_success:
   <ISO now>, last_status:'ok'})`. Print a one-line summary. Exit 0.
9. **On failure/timeout/guard-refusal**: `writeScheduleState(paths, name,
   {last_status:'error', last_error_at:<ISO now>})`, then **fail-loud** (below), then
   throw `WienerdogError` (exit 1).

**Fail-loud** (`async failLoud(paths, name, reason, logTail)`):

1. Best-effort email: spawn `wienerdog gws _alert` as a subprocess —
   `spawnSync(nodePath(), [wienerdogBin(), 'gws', '_alert', '--subject', <subject>,
   '--body', <body>], {env})`, where `<subject>` is `job <name> failed` and `<body>` is
   the reason plus the log tail. If it exits 0, done. (Works only when Google is
   configured + WP-018 merged.)
2. Else fall back to a **digest banner**: read `<paths.state>/digest.md` if it exists and
   prepend a single line at the very top, then write it back (atomic temp+rename):
   ```text
   > [!warning] Wienerdog job "<name>" failed at <ISO> — <reason>. See ~/.wienerdog/logs/<name>/.
   ```
   If `digest.md` does not exist, create it containing just that banner line. (The
   banner is transient — the next `dream`/`sync` regenerates `digest.md` — which is
   acceptable: its job is to surface the failure in the *next* session; documented
   accepted limitation.)

`logTail` = the last ~2 KB of the run's log file (best-effort; empty string if
unreadable). Never let a fail-loud error mask the original failure — wrap the whole
`failLoud` body so it cannot throw.

**`run-job --catch-up` logic:**

1. `jobs = listJobs(paths)`; `state = readScheduleState(paths)`; `now = new Date()`.
2. For each job, compute **today's scheduled fire time** `fire` = today at the job's
   `at` HH:MM (local). The job is **overdue** iff `now >= fire` AND
   (`state[name].last_success` is absent OR `new Date(last_success) < fire`). (Daily
   jobs; this catches "should have fired today but didn't because the machine was off".)
3. Run each overdue job **sequentially** by invoking the same `run-job <name>` code path
   (not a fresh subprocess — call the internal single-job runner). A job that succeeds
   updates its `last_success`, so the next hourly catch-up will not re-run it.
4. A single job's failure must not abort the rest: catch per-job, fail-loud for that
   job, continue. Exit 0 if all overdue jobs ran (even if some failed — their own
   fail-loud fired); print a summary of what ran.
5. On macOS, `--catch-up` is invoked by the catch-up plist WP-013 registers. On Linux
   it is effectively a no-op path (systemd `Persistent=true` already caught up) but the
   command still works if called.

## Implementation notes & constraints

- **Zero new runtime dependencies.** Node stdlib only (`child_process`, `fs`, `path`,
  `os`). JSDoc types, no TypeScript, no build step.
- **The watchdog must leave nothing behind (ADR-0004).** Reuse `dream.js`'s exact
  structure: detached child, `Promise.race([exit, timer])`, `process.kill(-pid,
  'SIGKILL')` on timeout, `finally { clearTimeout }`. Assert in a test (with a fake job
  command that hangs) that after a timeout there is no surviving child and no timer.
- **Clean env is explicit, never inherited wholesale** — the claude-os L5/L6 lesson. The
  test asserts the constructed env has an absolute `PATH` containing the node dir and
  `HOME`, and `WIENERDOG_JOB=<name>`, and does NOT carry arbitrary inherited vars beyond
  the small allowlist.
- **TCC-guard compares path segments**, not string prefixes (`Documents` ≠
  `DocumentsArchive`), and is darwin-only. Test both a protected path (refused) and a
  `$HOME`-root path like `~/wienerdog` (allowed), plus a non-darwin platform (always
  allowed).
- **Fail-loud must not throw.** Wrap the email + banner in try/catch so the original
  failure is always the surfaced error. Test the banner path directly (no gws
  configured → banner prepended to `digest.md`).
- **Testing without real schedulers or a real brain:** point the job at a **fake
  command** via a small seam — accept an injectable spawner (e.g.
  `run(argv, {spawn = child_process.spawn} = {})`) OR read a `WIENERDOG_RUNJOB_CMD` env
  override that replaces `claude`/`node` with a controllable fake script (mirror
  `WIENERDOG_DREAM_CMD` from WP-017). Prefer the env-override seam for parity with the
  dream tests. Tests then exercise success, non-zero exit, and hang → timeout without a
  real model or launchd.
- When uncertain: choose the simpler option and record it under "Decisions made". Do NOT
  expand scope (no Codex-exec routine brain, no Windows, no per-job env customization
  beyond the allowlist).

## Acceptance criteria

- [ ] `tccguard.guard` refuses a vault under `~/Documents`/`~/Desktop`/`~/Downloads`/
      `~/Library/Mobile Documents` on darwin, allows `~/wienerdog`, and always allows on
      non-darwin; segment comparison (no `DocumentsArchive` false match).
- [ ] `run-job dream` (fake command, success) writes `last_success`+`last_status:'ok'`
      to `state/schedule.json`, tees output to `logs/dream/<stamp>.log`, and exits 0.
- [ ] The clean env passed to the child has an absolute `PATH` including the node dir,
      `HOME`, and `WIENERDOG_JOB=dream`, and excludes non-allowlisted inherited vars.
- [ ] A hanging job hits the watchdog: the child tree is killed, `last_status:'error'`
      is recorded, fail-loud fires, `run-job` exits 1, and no child/timer survives.
- [ ] Fail-loud with no gws configured prepends the `> [!warning] ...` banner to
      `state/digest.md` (creating it if absent) and never throws.
- [ ] A TCC-guard refusal fails loud and exits 1 without spawning the brain.
- [ ] Log rotation keeps only the newest 14 `*.log` files in a job's log dir.
- [ ] `run-job --catch-up` runs a job whose `last_success` predates today's fire time
      (and `now >= fire`), skips one not overdue, updates watermarks, and does not abort
      the batch on a single job's failure.
- [ ] `npm test` and `npm run lint` pass.
